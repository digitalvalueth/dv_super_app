# v1.5.1.1 Fix Report

เอกสารนี้สรุปปัญหาที่เจอหลังแก้รอบ `1.5.1` ว่าเกิดจากอะไร และแก้ในรอบ `1.5.1.1` อย่างไร

## 1. หน้า Users ค้นหาชื่อจริงภาษาไทยไม่เจอ

สาเหตุ: ช่องค้นหาเดิมค้นเฉพาะ `name` และ `email` จึงไม่เจอข้อมูลที่อยู่ใน `fullName` เช่น `อั๋น เตะชลือ` รวมถึง `baCode`, `branchCode` และชื่อสาขา

วิธีแก้: ขยาย search index ของหน้า Users ให้รวม `fullName`, `baCode`, `companyName`, `branchCode`, `branchName` และ `branchNames` พร้อม normalize ช่องว่างและตัวอักษร

ไฟล์ที่แก้: `platform-web/app/stock-counter/dashboard/users/page.tsx`

## 2. Modal แก้ไขผู้ใช้เลื่อนยากและปุ่มอยู่ล่างจอ

สาเหตุ: Modal เดิมเป็นกล่องความสูงตาม content และไม่มี scroll container ภายใน เมื่อ field เยอะจะล้นหน้าจอ

วิธีแก้: ปรับ modal เป็น `max-h-[90vh]`, แยก header/body/footer, ให้ body scroll ได้ และให้ปุ่มยกเลิก/บันทึกอยู่ใน footer

ไฟล์ที่แก้: `platform-web/app/stock-counter/dashboard/users/page.tsx`

## 3. Import Users ดึงชื่อ/อีเมล/BA Code ไม่ครบจาก Excel

สาเหตุ: Parser อ่าน header แบบ exact key เช่น `email` เท่านั้น จึงพลาดหัวคอลัมน์แบบ `E-Mail`, `Full Name`, header มีช่องว่าง, BOM หรือสะกดต่างกันเล็กน้อย

วิธีแก้: เพิ่ม header normalizer สำหรับ import users รองรับ `E-Mail`, `fullName`, `baCode`, `branchCode`, `supervisorEmail`, `seller` และชื่อคอลัมน์ภาษาไทยบางรูปแบบ พร้อม preserve BA Code ที่มีเลข 0 นำหน้า

ไฟล์ที่แก้: `platform-web/app/stock-counter/dashboard/import-users/page.tsx`

## 4. Import Users เจอ user เดิมแล้วไม่ map ด้วย email

สาเหตุ: API ส่งคำเชิญเคยตอบ `409` เมื่อ email อยู่ในบริษัท/สาขาเดิม ทำให้ข้อมูลจากไฟล์ import เช่น `fullName`, `baCode`, `seller`, `supervisor` และ `branchIds` ไม่อัปเดตเข้า user เดิม

วิธีแก้: ปรับ API ให้ถ้าเจอ user เดิมในบริษัทเดียวกัน จะอัปเดต user doc เดิมโดยตรงจาก email และ map `branchIds`, `branchNames`, `sellerCategory`, `supervisorId/supervisorEmail` ให้ครบ

ไฟล์ที่แก้: `platform-web/app/api/invitations/send/route.ts`, `platform-web/app/stock-counter/dashboard/import-users/page.tsx`

## 5. Import Branches ไม่มี supervisorEmail และ seller บางเคสไม่ขึ้น

สาเหตุ: Template/Parser import branches ยังไม่มี `supervisorEmail` และอ่าน header แบบ exact key ทำให้ seller จากไฟล์บางรูปแบบไม่ถูกบันทึก

วิธีแก้: เพิ่ม `supervisorEmail` ใน template, preview และ Firestore write รวมถึง map supervisor จาก email ถ้ามี user อยู่แล้ว และบันทึกทั้ง `sellerCategory` กับ `seller` alias

ไฟล์ที่แก้: `platform-web/app/stock-counter/dashboard/import-branches/page.tsx`

## 6. Supervisor เป็น user ใหม่แล้วสาขายังไม่ผูกทันที

สาเหตุ: สาขาที่ import มาก่อนยังไม่มี `supervisorId` เพราะ supervisor user ยังไม่ถูกสร้าง/รับคำเชิญ และ invitation เดิมไม่ได้ auto match จาก `supervisorEmail`

วิธีแก้: ให้ invitation API ค้นสาขาที่มี `supervisorEmail` ตรงกับ email ของ supervisor/manager แล้วใส่เข้า `managedBranchIds`; เมื่อ user accept จะอัปเดตกลับไปที่ branch ด้วย `supervisorId`, `supervisorName`, `supervisorEmail`

ไฟล์ที่แก้: `platform-web/app/api/invitations/send/route.ts`, `platform-web/app/api/invitations/accept/route.ts`

## 7. Employee รับคำเชิญแล้วชื่อสาขาผิดจนต้องแก้เอง

สาเหตุ: Accept flow ใช้ `branchName` ที่ copy อยู่ใน invitation เป็นหลัก ถ้า invitation มีชื่อเก่าหรือว่าง user doc ก็จะได้ชื่อผิดตามนั้น

วิธีแก้: ตอน accept ให้ resolve branch จาก collection `branches` ด้วย `branchId` อีกครั้ง แล้วค่อยเขียน `branchName`, `branchCode`, `branchIds`, `branchNames`

ไฟล์ที่แก้: `platform-web/app/api/invitations/accept/route.ts`, `app/invitation/[token].tsx`, `components/notification-dropdown.tsx`

## 8. ผู้ใช้สาขาเดียวไม่เห็นชื่อสาขา/Stock Counter บางจุดใน Mobile

สาเหตุ: บางหน้าพึ่ง `branchId`/`branchName` เท่านั้น หรือแสดง branch tab เฉพาะกรณี `branchIds.length > 1`; ถ้า user doc มีข้อมูลอยู่ใน `branchIds`/`branchNames` แต่ field เดี่ยวว่าง จะถูกมองว่าไม่มีสาขา

วิธีแก้: เพิ่ม fallback เป็น `branchIds[0]` และ `branchNames[branchId]` ในหน้า Home, Profile และ Stock Counter products พร้อมแสดง branch chip แม้มีสาขาเดียว

ไฟล์ที่แก้: `app/(tabs)/home/index.tsx`, `app/(tabs)/settings/index.tsx`, `app/(mini-apps)/stock-counter/products/index.tsx`

## 9. แก้ไขงานนับสต็อกตามแบรนด์แล้วบันทึกสินค้าทั้งหมด

สาเหตุ: ปุ่มแบรนด์สินค้าในหน้าแก้ไขงานสาขาเป็นแค่ตัวกรองหน้าจอ ทำให้เมื่อเลือก `PrimaNest` ฝั่งขวาแสดง 32 รายการ แต่ `assignmentForm.productIds` ยังเก็บสินค้าทุกแบรนด์ที่ถูกซ่อนไว้ พอกดบันทึกจึงเขียนสินค้าทั้งหมดกลับเข้า assignment

วิธีแก้: ทำให้ brand filter เป็น scope ของการบันทึกจริง ถ้าเลือก `PrimaNest` จะบันทึกเฉพาะ productIds ของ `PrimaNest` ตามจำนวนที่แสดง และปรับตัวเลข footer/ปุ่มลบทั้งหมดให้ตรงกับแบรนด์ที่กำลังเลือก

ไฟล์ที่แก้: `platform-web/app/stock-counter/dashboard/branches/[id]/page.tsx`

## 10. แก้ไขผู้ใช้เลือกหัวหน้างานคนละสาขาได้

สาเหตุ: Dropdown `Supervisor (หัวหน้าพนักงาน)` ในหน้าแก้ไขผู้ใช้ดึงหัวหน้างานทุกคนในบริษัท แต่ไม่ได้โหลด `branchIds`/`managedBranchIds` ของหัวหน้างานมาเทียบกับสาขาของพนักงาน จึงเลือกหัวหน้างานที่อยู่หรือดูแลคนละสาขาได้

วิธีแก้: เพิ่มข้อมูลสาขาของหัวหน้างานใน dropdown, แสดงเฉพาะหัวหน้างานที่มีสาขาตรงกับพนักงาน และ validate ซ้ำก่อนบันทึก ถ้าหัวหน้างานเดิมคนละสาขาจะขึ้นคำเตือนให้เลือกใหม่หรือเคลียร์ค่า

ไฟล์ที่แก้: `platform-web/app/stock-counter/dashboard/users/page.tsx`

## 11. ขอบขาวด้านล่างใต้ Tab Bar บนมือถือ

สาเหตุ: Mobile tabs ใช้ `NativeTabs` แบบ iOS scroll-edge โปร่งใส (`disableTransparentOnScrollEdge={false}`) และหน้า Home ครอบ `SafeAreaView` แบบ default ที่รวม bottom safe area ด้วย ทำให้เฉพาะหน้าแรกมีพื้นหลังด้านล่างซ้อนกับ tab bar ต่างจากหน้า Profile ที่ใช้เฉพาะ top safe area

วิธีแก้: กำหนด background ของ `NativeTabs` ให้ใช้สีพื้นหลังแอปบน iOS, ปิด transparent scroll-edge และปรับหน้า Home ให้ใช้ `edges={["top"]}` พร้อมย้ายระยะเผื่อด้านล่างไปไว้ใน `ScrollView` แทน blank view

ไฟล์ที่แก้: `app/(tabs)/_layout.tsx`, `app/(tabs)/home/index.tsx`

## 12. เลือกสาขาที่รับผิดชอบยากเมื่อมีหลายร้อยสาขา

สาเหตุ: Modal แก้ไขผู้ใช้เดิมแสดงสาขาเป็น checkbox list เดียว ถึงจะมี search แต่เมื่อบริษัทมี 500-600 สาขาจะดูยากว่าเลือกอะไรไปแล้ว และต้องไล่หาใน list เดิมตลอด

วิธีแก้: ปรับ `สาขาที่รับผิดชอบ` เป็น selector 2 ฝั่ง โดยฝั่งซ้ายแสดง `สาขาที่เลือกแล้ว` พร้อมค้นหา/ลบ/ล้างทั้งหมด และฝั่งขวาแสดง `สาขาที่ยังไม่ได้เลือก` พร้อมค้นหา/เพิ่มทีละสาขา/เพิ่มผลลัพธ์ที่ค้นหา รวมถึงยังคง validate หัวหน้างานตามสาขาที่เลือกเหมือนเดิม

ไฟล์ที่แก้: `platform-web/app/stock-counter/dashboard/users/page.tsx`

## 13. เปิดสิทธิ์ Watson Excel Validator แล้วกดเข้าไม่ได้

สาเหตุ: หน้าเลือก Module และหน้าจัดการสิทธิ์อ้างอิง module จาก Firestore แต่หน้า Watson ใช้ `ModuleGuard moduleId="watson-excel-validator"` โดยตรง ใน production พบว่า module doc id และค่าที่บันทึกใน `users.moduleAccess`/`companies.enabledModules` เป็นชื่อแสดงผล `Watson Excel Validator` ทำให้ card แสดงว่าเข้าได้ แต่พอเข้า route แล้ว guard ตรวจคนละ id และเด้งกลับหน้าแรก อีกกรณีคือ guard ใช้ `userData.companyId` เท่านั้น ทั้งที่หน้าเลือก Module ใช้ `activeCompanyId` สำหรับผู้ใช้หลายบริษัท

วิธีแก้: เพิ่มการตรวจสิทธิ์แบบ normalize module id ให้ `Watson Excel Validator`, `watson` และ `watson-excel-validator` เทียบกันได้ทั้งหน้าเลือก Module, route guard และตารางจัดการสิทธิ์, ปรับ `ModuleGuard` ให้ใช้ `activeCompanyId || userData.companyId` เหมือนหน้าแรก และให้ guard ดึง user doc ล่าสุดจาก Firestore ด้วย uid/email ก่อนปฏิเสธสิทธิ์ เพื่อรองรับเคสเพิ่งเปิดสิทธิ์แล้ว session ยังเห็นค่าเก่า

ไฟล์ที่แก้: `platform-web/lib/module-service.ts`, `platform-web/components/guards/module-guard.tsx`
